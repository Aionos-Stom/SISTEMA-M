1. Descripción General del Sistema

El Sistema M — Peravia es una plataforma web de gestión territorial diseñada para la Administración de la Provincia de Peravia, República Dominicana. Permite registrar, organizar y consultar información de ciudadanos y simpatizantes distribuidos por municipios, zonas, sectores y mesas electorales.

El sistema opera completamente en el navegador (sin instalación) y se conecta a una base de datos en la nube mediante Supabase. Cuenta con un modo demo que funciona sin conexión a internet para pruebas y presentaciones.

Objetivos Principales

📊 Gestión Territorial

  • Registro de ciudadanos por zona geográfica
  • Organización por municipio, sector, mesa y recinto
  • Estadísticas en tiempo real por área
  • Filtros avanzados de búsqueda

👥 Control de Acceso

  • 5 niveles de roles jerárquicos
  • Aprobación de usuarios por administrador
  • Visibilidad de datos según rol
  • Auditoría completa de acciones

📤 Exportación

  • Exportar registros a Excel (.xlsx)
  • Generación de reportes PDF
  • Filtros aplicados en la exportación
  • Datos organizados por columnas

🔒 Seguridad

  • Autenticación con Supabase Auth
  • Políticas RLS en base de datos
  • Registro de auditoría inmutable
  • Recuperación de contraseña por email
Municipios cubiertos: Baní · Nizao · Los Cacaos · Sabana Buey · Matanzas · Villa Fundación · Paya · Santana · Pueblo Viejo

4. Roles y Permisos

El sistema implementa una jerarquía de 5 roles. Cada rol hereda las capacidades del nivel inferior y agrega las propias.

👑

Administrador — Nivel 5

Acceso total al sistema. Puede ver todos los registros de todos los municipios, gestionar usuarios (aprobar, rechazar, editar, eliminar), ver el log de auditoría completo y exportar cualquier dato.

🏛️

Coordinador Municipal — Nivel 4

Ve todos los registros de su municipio (provincia) asignado. Puede editar y eliminar registros de su área. No accede a auditoría ni gestión de usuarios.

🔍

Supervisor de Zona — Nivel 3

Ve únicamente los registros que él mismo ha creado. Puede editar registros propios. No puede eliminar ni gestionar usuarios.

✏️

Registrador — Nivel 2

Puede crear nuevos registros ciudadanos y ver solo los que él registró. No puede editar ni eliminar registros existentes.

👁️

Observador — Nivel 1

Solo puede ver los registros que él mismo creó. No puede crear, editar ni eliminar. Acceso de solo lectura.

⚠️ Importante: Los usuarios nuevos quedan en estado pendiente hasta que un Administrador los apruebe desde el panel de Gestión de Usuarios. Sin aprobación, el login es bloqueado con mensaje de error.

Recuperación de Contraseña

El usuario ingresa su correo en el formulario "¿Olvidé mi contraseña?". Supabase envía un enlace de recuperación al email. Al hacer clic en el enlace, el sistema detecta el parámetro type=recovery en la URL y abre automáticamente el modal para establecer una nueva contraseña.

Gestión de Sesión

La sesión se persiste en localStorage con la clave peravia-auth-token. Al recargar la página, el sistema verifica si hay sesión activa y carga el dashboard automáticamente sin necesidad de volver a iniciar sesión. El token se renueva automáticamente antes de expirar.

8. Módulo de Auditoría

Registro inmutable de todas las acciones realizadas en el sistema. Solo visible para el Administrador. No se puede modificar ni eliminar ningún registro de auditoría.

Acciones registradas automáticamente

Código de acciónDescripción
SESSION_LOGINInicio de sesión exitoso
SESSION_LOGIN_FAILEDIntento de login fallido (registra el usuario/email intentado)
SESSION_LOGOUTCierre de sesión
PASSWORD_RESET_REQUESTSolicitud de recuperación de contraseña
PASSWORD_RESET_COMPLETEContraseña actualizada exitosamente
VOTER_CREATENuevo registro ciudadano creado
VOTER_EDITRegistro ciudadano editado
VOTER_DELETERegistro ciudadano eliminado
VOTER_DUPLICATEIntento de registrar una cédula ya existente
USER_EDITPerfil de usuario modificado
USER_APPROVEUsuario aprobado por administrador
USER_REJECTUsuario rechazado y eliminado

Datos registrados por cada acción

Cada entrada de auditoría guarda: ID del actor, nombre del actor, rol del actor, código de acción, objetivo (ID o nombre del elemento afectado), detalles descriptivos y fecha/hora exacta.

Filtros de Auditoría

El administrador puede filtrar el log por: texto libre (nombre, objetivo, detalles), tipo de acción, actor específico, y rango de fechas (desde / hasta).

10. Seguridad — RLS y Políticas

Row Level Security (RLS) es una característica de PostgreSQL que filtra automáticamente las filas que cada usuario puede ver o modificar, directamente en la base de datos — independientemente de lo que haga el código del cliente.

Funciones de seguridad (SECURITY DEFINER)

FunciónRetornaUso
get_mi_rol()TEXTRol del usuario autenticado actual
get_mi_estado()TEXTEstado (aprobado/pendiente/rechazado) del usuario actual
get_mi_provincia()TEXTMunicipio asignado al usuario actual
get_mi_nivel()INTNivel numérico del rol (1-5)
get_email_from_username()TEXTBusca email por username (para login, accesible sin sesión)
check_username_exists()BOOLEANVerifica si un username ya existe (para registro)

Política consolidada para registros (SELECT)

La política registros_select (aplicada por supabase_patch.sql) permite ver un registro si se cumple alguna de estas condiciones:

1. El usuario es Administrador
2. El usuario es Coordinador o superior Y el registro es de su municipio
3. El usuario está aprobado Y fue quien creó el registro (registrado_por_id = auth.uid())

Resumen de políticas por tabla

TablaOperaciónCondición
registrosSELECTAdmin ó (Coord+ en su provincia) ó (propio registro y aprobado)
registrosINSERTNivel ≥ 2 (Registrador+) y estado aprobado
registrosUPDATENivel ≥ 3 (Supervisor+) y estado aprobado
registrosDELETENivel ≥ 4 (Coordinador+) y estado aprobado
usuariosSELECTUsuario aprobado ó propio perfil ó Admin
usuariosINSERTSolo su propio perfil (auth_user_id = auth.uid())
usuariosUPDATEAdmin (cualquiera) ó propio perfil
usuariosDELETESolo Admin
auditoriaSELECTNivel ≥ 4 (Coordinador+) y aprobado
auditoriaINSERTUsuario aprobado ó logins fallidos (anon)
auditoriaUPDATE/DELETEBloqueado — nadie puede modificar el historial

12. Exportación de Datos

Exportar a Excel (.xlsx)

Genera un archivo Excel con los registros actualmente filtrados en la tabla. Usa la librería SheetJS. Las columnas exportadas son: Nombre, Cédula, Teléfono, Municipio, Región, Sección, Distrito, Zona, Sector, Mesa, Recinto, Observación, Registrado por, Rol del registrador, Fecha de registro.

El nombre del archivo incluye la fecha de exportación: registros_peravia_YYYY-MM-DD.xlsx

Exportar a PDF

Genera un PDF con los registros filtrados usando jsPDF con el plugin autoTable. Incluye encabezado con el nombre del sistema, fecha de generación y total de registros. Las columnas se ajustan automáticamente al ancho de la página.

💡 Tip: Los filtros aplicados en la tabla se respetan en la exportación. Si filtras por municipio "Baní" antes de exportar, el archivo solo contendrá registros de Baní.

13. Configuración e Instalación

Paso 1 — Crear proyecto en Supabase

Ir a supabase.com, crear una cuenta y un nuevo proyecto. Anotar la Project URL y la clave anon/public desde Settings → API.

Paso 2 — Configurar credenciales en Script.js

Abrir Script.js y reemplazar las líneas al inicio del archivo:

const SUPABASE_URL = 'https://tu-proyecto.supabase.co';
const SUPABASE_ANON = 'tu_clave_anon_aqui';

Paso 3 — Crear las tablas en Supabase

En Supabase → SQL Editor → New Query, pegar y ejecutar el contenido completo de supabase_setup.sql. Esto crea las 3 tablas, índices, funciones de seguridad y políticas RLS.

Paso 4 — Aplicar el parche de RLS

Ejecutar también el contenido de supabase_patch.sql. Esto consolida las políticas SELECT de la tabla registros para que todos los roles puedan ver sus propios registros correctamente.

Paso 5 — Abrir la aplicación

Abrir Index.html directamente en el navegador (doble clic) o servirlo con cualquier servidor web estático. No requiere Node.js, Python ni ningún servidor de aplicaciones.

Paso 6 — Crear el primer usuario Administrador

Registrarse desde el formulario de registro seleccionando el rol Administrador. El primer usuario puede aprobarse a sí mismo directamente en Supabase → Table Editor → tabla usuarios → cambiar estado a aprobado. A partir de ahí, el Administrador puede aprobar a los demás usuarios desde el panel.
